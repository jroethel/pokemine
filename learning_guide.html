Design-process learnings

A resource inventory in the brief is a request to map the solution space, not background color.
Jeremy listed "Google Pro AI plan" in his stack. The session evaluated only API-based image generation and never surfaced the implied alternative: tapping the consumer plan's already-paid image quota via browser automation. Jeremy had to propose that direction (Plan B) himself.
Direct feedback, 2026-07-12: "capture the learning that didn't have you ask me about the browser idea when I mentioned I had AI Pro. I'm expecting you to catch things like that!" Future sessions: walk the inventory item by item and ask "what approach does this asset uniquely enable?" - surface every viable option with a recommendation, including options you would argue against.
Verify provider capabilities with live calls during design, not during build.
Two one-minute curl probes changed the design's facts: the Gemini free tier returns quota limit: 0 for gemini-3.1-flash-image (billing required, ~$0.034/image), and Z.AI's glm-image exists but the GLM Coding Plan excludes image generation (error 1113, needs balance). Text on gemini-flash-latest verified free and working.
The initial recommendation ("a free AI Studio key covers images") was wrong and was caught by checking before it shaped the spec. Web research alone had already gone stale.
A rival "Plan B" is often just a component of Plan A wearing an architecture costume.
The browser-bridge idea looked like a competing app design. Reframed, it is a fourth image provider behind the same generate() interface, fulfilled out-of-band via a job folder (bridge-jobs/*.json in, *.png out). Plan A absorbed its good parts (Drive-synced DATA_DIR, the job contract) without building the fragile part.
Gitignore secrets before the first commit, not after.
nano-api.md holds a live API key and sat in the repo root when git init ran. The .gitignore (covering nano-api.md, .env, data/) went in with the first commit, so the key never entered history.
