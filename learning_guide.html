Build execution learnings (2026-07-12, build day)

Route work by the One-Minute Test before spinning up machinery.
The 8-task implementation plan looked "agent team"-shaped but scored as ONE AGENT: strictly sequential tasks, exceptional checkability (plan embedded complete code plus expected outputs), judgment already spent at design time. One Opus worker executed all 8 tasks in ~23 minutes for ~$0.14 of API spend; a subagent-per-task team would have added review theater on top of a test suite already doing the reviewing.
Model routing rule honored all day: Fable for architecture, probing, arbitration, and validation; Opus for implementation; Haiku for mechanical fetches.
Never accept a worker's self-report; validate against artifacts.
Every agent completion was gated the same way: rerun npm test personally, read git log for commit hygiene and secret leakage, view the actual screenshots, re-derive one number (63x88mm = 238.1x332.6px at 96dpi). All workers were honest today; the gate is what makes that a finding instead of an assumption.
Verify the risky contract BEFORE building on it, with the cheapest possible probe.
Before the bridge was built, three live probes against the real signed-in Gemini web app established the whole DOM contract: execCommand('insertText') works untrusted, the Send button only renders once input exists, images land in ~18s as blob URLs, and canvas re-encode extracts them where fetch cannot. The builder agent then coded against verified facts instead of guesses.
The same discipline earlier: two 3-cent API probes pinned "billing works", "response is image/jpeg not PNG", and "aspectRatio 4:3 accepted" before the plan was written.
When a transport keeps failing, change strategy instead of escalating.
Getting image bytes out of the browser failed three ways: macOS TCC blocked ~/Downloads, the signed download URL dead-ended at an auth wall, and Brave's Private Network Access blocked page-to-localhost fetches even with correct CORS+PNA headers. The fourth path shipped: Gemini's Copy-image button, through the macOS clipboard via osascript, into the file job queue. Three strikes on one approach means the approach is wrong, not the execution.
Real finding inside the failure: PNA blocking means the extension's service-worker proxy is architecturally REQUIRED for page-to-localhost, not a convenience.
A multi-driver contract proves itself when the dumbest driver works.
Socko #003 was fulfilled by a human-grade driver (clipboard and file copy) with zero code changes, confirming the bridge job queue genuinely accepts any driver - extension, agent, or hand.
One agent owns one file; art direction and asset generation stay with the orchestrator.
The background-image request arrived mid-flight and was folded into the already-dispatched restyle agent's brief rather than spawning a second writer for style.css. The orchestrator generated and chose the logo and background art (taste calls), agents wired them in (implementation).
Flagged blemishes get fixed at the gate, not filed away.
The polish agent flagged bare-string moves rendering an empty card area but left it out-of-scope. The fix (shape validation inside the retry net, ~10 lines plus test) happened at validation time under "make appropriate changes" authority - small enough to do, real enough for a kid to hit.

Context engineering learnings (2026-07-12)

Separate refreshable knowledge from code; inject it, don't bake it.
Pokemon canon lives in NotebookLM (Jeremy curates) -> scripts/refresh-notebook-cache.py (nlm CLI, no MCP round-trips) -> distilled canon-pack.md -> appended to the text-gen system prompt at call time via CANON_FILE. Adding lore never touches code: refresh, re-distill, restart.
Proof it works: Socko's backstory arrived as a Johto folk legend through the bridge path.
Design tokens beat vibes when handing aesthetics to an agent.
The grand-reveal restyle worked because the brief was concrete: named hex tokens from two fetched color standards, the handbook cover distilled to CSS-reproducible motifs (45-degree lattice, sticker outlines, chunky stroked type), per-view direction, and hard constraints (print dimensions frozen, no deps). The agent's two deviations were both defensible because the brief made intent legible.
Cache what you'll re-read; index what you cached.
17 notebook sources cached locally (~750KB, gitignored, regenerable), distilled twice into committed docs. Two Bulbapedia sources turned out to be Cloudflare-block stubs - caught because the cache made contents inspectable.

Live-iteration learnings (2026-07-12 evening, first real users)

Real users find the gaps that specs and tests cannot.
Within the first play session: a rename typed into the redraw box got painted INTO the artwork as text (fix: a no-text rule in every image prompt, plus placeholder copy steering "Change its LOOK"); the zai provider sat in the dropdown as a loaded trap (no balance = guaranteed friendly error); and the measured 1-in-4 JSON flake produced a real double-flake "escaped!" in the wild (fix: third parse attempt, free).
Every one of these passed all 16 tests. Tests verify the contract you thought of; kids execute the contract you didn't.
Separate content from code so delight can scale without engineering.
The loading messages kids loved went from 6 hardcoded strings to 160 canon-flavored phrases in a standalone public/loading-messages.js - same pattern as the canon pack: content-as-data, editable by anyone, no app-code changes to extend. Random starting index so variety shows immediately.
Feature requests arriving mid-agent-run get routed by file ownership.
Seven additions landed while one agent was mid-pass. Rule held throughout: whoever owns the file makes the change. Agent owned server.js/app.js (got the queued additions via messages); the orchestrator owned lib/text.js and new content files (edited directly, commits held until the agent released the repo to avoid git index races).
Verify before accusing: the orchestrator pattern-matched a failure to an innocent worker.
When the kids' Pokedex "lost" entries mid-play, the orchestrator fired an urgent stop-order at the feature agent, blaming its Release/Redraw testing - without checking evidence first. The agent proved innocence with receipts: it had unprompted used a throwaway /tmp data dir on a separate port and never clicked the suspect buttons. Actual causes: a dead/stale server process, a mock-artist selection, and probable Drive streaming stalls. The data was intact the whole time.
The worker doing the right thing unprompted deserved better than a pattern-matched accusation. Diagnose from artifacts (ls, ps, curl) BEFORE assigning cause - the same evidence-first rule that applies to code applies to blame.
The friendly error is UX; the console error is forensics; neither is a log.
"The Pokemon escaped!" plus console.error worked until the terminal closed. Durable one-line entries now append to the Drive-synced errors.log, and the cost ledger got the same treatment (costs.json: session and all-time, per provider) after the browser-side counter proved forgettable.

Plan-audit learnings (2026-07-14, evolutions & stages)

Setup: another model wrote the implementation plan for the evolutions/stages feature. Fable's job was to audit it against the real code before anything got built. The audit changed the plan materially; these are the transferable lessons.

Audit a plan by re-deriving its load-bearing claims, not by reading for plausibility.
The plan's centerpiece CSS fix ("bump padding 3em to 3.3em") read fine and was mathematically broken: .card-eyebrow has font-size: .62em, so its 3em padding resolved to 1.86 card-em while the sibling row got a true 3 card-em - THE reported overlap and differential-shift bug. The 3.3em bump would have moved the eyebrow to 2.05em, still under the 2.5em badge. Five minutes of arithmetic replaced a tune-it-in-browser guess with a deterministic fix (one shared .card-top wrapper inset).
A plan review that only checks structure and coverage approves confident wrongness. Every number that decides a fix gets recomputed from the source files.
The audit cuts both ways: the reviewer erred too, and so did the rebuttal.
Fable's review cited the text.js retry comment as evidence of instruction-drift; it is actually about malformed JSON - loose framing, right conclusion. The other model's otherwise-gracious rebuttal then claimed the calc fix was "borderline tight" using content-box arithmetic in a * { box-sizing: border-box } stylesheet. Both errors were caught the same way the plan's error was: by checking the file instead of the prose.
Mutual audit converged on a plan neither model wrote alone. Neither side's confidence was evidence at any point.
Check the plan against the user's literal spec, not just against the code.
Jeremy asked for the dex sorted "A1, A2, A3, B1..." alphabetically. The plan silently kept owner-rank as the primary sort key and alphabetized within groups - a plausible-looking deviation that would have shipped. Silent scope drift hides best inside otherwise-correct plans.
A plan that changes a contract must name the tests it breaks.
Adding backstory to STAGE_FIELDS turned an existing green test red (the mocked stage lacked the field). Neither the plan nor the first review caught it; the counter-review did. Now a standing check: for every contract change, grep the test fixtures that instantiate that contract.
Enforce invariants in code; prompts are suggestions.
The plan carried the per-stage HP/damage bands only in prompt prose, in a codebase whose own comments document the text model flaking ~1 in 4 calls. One drifted roll (a normal stage 3 at HP 240) would have made the 0.5% Mega jackpot look ordinary. A 10-line server-side clampStage made the tier system deterministic.
Same family as the move-shape validation from build day: validate/clamp at the boundary, never trust generation to obey ranges.

The full loop held: review, counter-review, merged plan, route, one agent, adversarial gate, taste gate.
loop-which scored the merged plan ONE AGENT (coupled full-stack change, superb checkability, judgment already spent). The plan said a forced Mega should show "stats above 200", but the locked design only raised the ceiling. Reliability-generation learnings (2026-07-15/16, faster & kinder build)

Setup: a detailed plan to make card generation rarely fail and never empty-handed (streaming Poke Ball progress, placeholder-art fallback, friendly errors, a server-driven bridge deadline, a pluggable text model). loop-which scored it ONE AGENT; loop-drive drove a single Opus worker; the orchestrator gated.

A plan with the judgment already spent routes to ONE AGENT, even at five coupled tasks across ten files.
loop-which scored this plan ONE AGENT despite its size. The decisive questions were checkability (every task shipped a failing-test-first step, so checking was cheaper than producing) and judgment (the plan embedded exact code blocks and line numbers; the worker executed, it did not design). The tasks also chained strictly: T3 consumed T1's provider threading and T2's placeholder export; T5 consumed T3's SSE events and T4's error box. So there was no parallelism a team could exploit, and the producer/critic split had already happened when the plan was written.
Size alone never earns a team. The route turns on independence, separation, and checkability; a big sequential plan with embedded checks is one worker's job.
When a worker adapts an existing test to a new contract, audit assertion-equivalence, not just green.
Converting POST /api/pokemon from one JSON blob to an SSE stream broke the existing lifecycle tests at the body-parsing layer. The Opus worker, with no instruction to, updated test/pokemine.test.js to parse SSE: it swapped r.json() for a helper that pulls the done event's record. The suite went green. The right gate question was not "did it pass" but "did the adaptation preserve what the test checks": it did, because only body extraction changed and every assertion on body.id and the lifecycle was untouched. A looser adaptation (asserting less, or swallowing the error event) would also have been green.
A green suite after a test-file deviation proves nothing about whether the test still tests the same thing. Diff the assertions, not the pass count.
A deviation's stated justification is itself a claim to re-derive.
The bridge-fulfillment test gained a POST /api/bridge/ping before create, justified as "to satisfy the new bridge-offline guard". Plausible, and the suite was green. But green does not tell you the test passes for the right reason. A grep of server.js confirmed /api/bridge/ping is a pre-existing route (the real extension's heartbeat) that sets the bridge.lastSeen the new guard reads, so the ping genuinely registers presence and the create genuinely clears the guard, instead of the test slipping past a guard that never fires.
Tests that pass for the wrong reason are worse than failing tests, because they look like evidence. The worker's narrative of why it made a change is a hypothesis; verify it against the code.
The device contract is a verification lane the agent cannot reach; hand it to the human, don't auto-decide it.
The plan's two open risks are both off-limits to a server-side executor: whether the Chromebook's browser fetch streams a POST response body (SSE-over-POST), and whether the real Gemini DOM still matches the bridge's image-detection heuristic. The unit tests prove the server emits a correct SSE stream through Node's fetch; they cannot prove a Chromebook renders it. The run stopped before both and surfaced them as decisions instead of guessing (it did not, for example, preemptively add the blocking-JSON fallback flag).
A TDD plan's tests cover the server contract. The client and hardware contract is a separate lane; auto-deciding it is the most common way a green run ships a broken feature.
Tier availability named the run-card's executor; it did not change the verdict.
Only the Strong tier (Opus 4.8 / GLM 5.2) was available this session. The ONE AGENT verdict was tier-robust because it needs exactly one solid executor and nothing to parallelize, so loop-drive ran with a Strong model orchestrating and executing rather than the usual Frontier orchestrator. The tier shortage did rule out AGENT TEAM (no Frontier critic, and no parallelism to spend a team on anyway), and it made GLM's mid-session 429 a real scheduling fact: Opus carried the long T3 and T5 pushes.
Ask the user what they have, but recognize when the answer changes the brief and not the route. Here the live question was execution flavor (drive now vs. just the verdict), not the abstract tier list.

Bridge-reliability & canon-data-flow learnings (2026-07-18)

canon-pack.md rides the system prompt for every text generation - creatures and trainers both - and is lazy-read so edits land on the next call.
The data flow worth capturing. CANON_FILE (docs/reference/canon-pack.md) is appended to the base SYSTEM prompt by buildSystem() (lib/text.js:13), and every callJSON() call uses it - so newPokemon (stage 1 / stageShape), evolvedStage, AND trainerBackstory all see the canon. It is read fresh on each call (comment at line 12), so editing canon-pack.md takes effect on the very next generation with no server restart. stageShape itself only hardcodes the 18 Pokemon types as the allowed types list; the myth, Pokedex voice, regions, and tone reach the model through the system prompt, not through the shape.
Surfaced 2026-07-18 answering "does the text prompt / stageShape use pokemon canon cached from the NotebookLM resource?" Deepens the 2026-07-12 "inject, don't bake" entry with the exact call sites and the lazy-read property.
FINISHER_MOVES (the NotebookLM canon-moves cache) feeds trainers only, not creatures.
The other NotebookLM-sourced artifact in lib/text.js - canon moves with power > 150 (pokemondb, cached 2026-07-13) - is injected only into trainerBackstory's prompt, never into newPokemon/stageShape. Creatures get universe canon; trainers additionally get the canon-moves constraint. Knowing which path consumes which cache matters when refreshing sources or debugging "why did the model produce X".
Refresh workflow: NotebookLM -> scripts/refresh-notebook-cache.py -> re-distill canon-pack.md and pokemon-design-notes.md.
The raw cache lives at docs/reference/notebook-cache/ (21 entries, gitignored, regenerable). The refresh script pulls via the nlm CLI. Distillation into canon-pack.md is a human/AI edit; the file header carries the source list and the distillation date. pokemon-design-notes.md (8.7KB) is the companion for deeper design detail that stageShape references ("details in pokemon-design-notes.md") but does not auto-inject.
A backgrounded driver tab silently stalls the bridge; the fix is a dedicated always-visible window, not per-job activation.
The bridge driver loop (setInterval/sleep in content.js) gets timer-throttled when gemini.google.com is backgrounded, and Gemini itself pauses inference on visibilitychange - so jobs stalled until the tab was manually activated, then timed out with no matched error pattern. Root-caused by reading content.js against Chrome's documented background-throttling, confirmed against Jeremy's live observation (active tab = fast; backgrounded = hangs). Fix: the service worker owns a dedicated popup window that stays visible (unthrottled) and hosts the ONLY content script allowed to drive jobs (an am-i-driver gate), so a throttled normal gemini tab cannot claim a job and stall on it.
macOS gotcha surfaced in the same debug: Chrome throttles fully-occluded windows too, so the driver window needs a sliver visible, not merely "not minimized". Jeremy runs maximized windows, so this is a live operating constraint.
"The pokeball stopped animating" was a fixture artifact, not a regression - mock art returns in <1ms.
Reported symptom: the Ultra Ball no longer appeared after the text phase. curl -sN of the running server showed both phase events (text/image) sent correctly on the wire; generation.log showed provider=mock, imageMs=0/1. The ultra phase was being set for about one millisecond - imperceptible. The code was working; the test provider was instant. The real fix was a 600ms minimum hold per phase so the animation reads with mock or fast art, not a logic change.
Reinforces the 2026-07-12 "verify the wire bytes" and the "reproduce before fixing" rules: capturing the SSE stream with one curl exonerated the server immediately, and the log's imageMs named the true cause. Treat the provider/fixture as a suspect before treating the symptom as a regression.
Jeremy wants manual control over auxiliary windows; never auto-open UI he did not ask for.
Three consecutive pushes in one session, all in the same direction: gate the close-reopen on server liveness ("only reopen if Pokemine is actually running"), then stop auto-opening on Brave start and extension reload, then add a toolbar "Launch driver window" button so the window opens ONLY on his click. The close-while-server-running reopen was kept as a single safety net (an accidental close mid-session should not silently break the kids' next generation). Direction of travel was consistently "I control when this exists."
Preference to carry forward: when a feature creates persistent UI (windows, popups, notifications, overlays), default to on-demand, not unsolicited. Surface it as a question if the auto behavior seems necessary.
Title-keep via interval, not MutationObserver: SPAs replace the <title> element, which orphans an observer.
To make the driver window findable in the dock / cmd-Tab / Brave Window menu (it otherwise shows as "Gemini", same as any gemini tab), the content script re-asserts document.title = 'Pokemine Bridge' every second. An interval over a MutationObserver because Gemini's SPA sometimes replaces the <title> element entirely, leaving an observer attached to a dead node. Boring and bulletproof beats event-driven and fragile for a one-string re-assertion in an always-visible tab.
LAN IPs drift; do not hardcode them in handoff docs.
whats_next.md told the Chromebook to hit 192.168.1.159, but the Mac was at .116 today - the router had reassigned it overnight. Nothing was wrong with the server. The doc now says "the Mac's current LAN IP (ipconfig getifaddr en0; re-check, it drifts)". Any handoff or run instruction that names a LAN IP should carry the re-check instruction, not a hardcoded octet.
